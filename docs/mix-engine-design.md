
# 4-Deck Mix Playback Engine — Final Design Doc (post-review)

> Historical note (#292, 2026-07-24): `MixNowPlayingInfo`, `nowPlayingStream`, and
> `loadSequentialQueue` described below were removed from `PlaybackEngine` as dead
> alternate authorities; `PlaybackSnapshot` via `QueueTimelineController` is the
> only now-playing surface. Read those sections as design history.

## 0. Grounding: what exists today (unchanged from draft, re-verified against source)

- `AudioPlayerService` (`client/lib/core/audio/audio_player_service.dart`) wraps ONE `just_audio.AudioPlayer` + `ConcatenatingAudioSource`. **Fully replaced.**
- `PlaybackState` (`client/lib/core/audio/playback_state.dart`) is a `ChangeNotifier` and the **only** UI-facing contract. Verified real surface includes `positionStream: Stream<Duration>`, `currentMediaItemStream: Stream<MediaItem?>`, `playerStateStream: Stream<PlayerState>` (just_audio type, consumers check `.processingState == ProcessingState.completed`), `currentIndexStream`, plus `isPlaying/position/bufferedPosition/duration/currentItem/queue/currentIndex/shuffleEnabled/loopMode/playbackContext` and the play/seek/queue mutator methods. **Confirmed consumer**: `PlayRecorderService` (`client/lib/core/audio/play_recorder_service.dart`) subscribes to exactly `currentMediaItemStream`, `positionStream`, and `playerStateStream` (for `ProcessingState.completed`) and reads `duration`/`playbackContext` synchronously — this is load-bearing for feature C15 (one play per continuous listen) and drives `PlayRecordDecider.onCompleted()`/`onPosition()`. **The original draft's §6 claim that "this contract does not change" was incomplete** — it omitted these three streams. This doc corrects that (§6, §1.9).
- `StackedWaveformTimeline` (prototype), `TimelineClip`/`TrimRange` (pure math, kept), `QueueProvider` (already treats the queue as a trivial mix plan via `_queueTimingClips()`), `PlaybackCacheManager`, `SignedAudioUrlService` + `PlaybackSourceResolver` — as in the original draft.
- **Verified**: `PlaybackSourceResolver.resolveQueue(List<Map<String,dynamic>>)` batches local-artifact lookup → signed-descriptor batch → cache lookup → builds `MediaItem`s, with `_scheduleWarm(cache, remoteIds, cachePaths, descriptors)` bounded by `cachePrefetchLimit` (default 3) and a `protect` set of in-use cache ids. It depends on `LocalAudioArtifactResolver?`, `PlaybackCacheManager?`, `SignedAudioUrlService`, **not** a per-id resolve API. The original draft's `DefaultEngineAudioSourceResolver(sourceResolver: PlaybackSourceResolver, cacheManager, signedAudioUrlService)` stub does not actually fit this shape (batch/Map API vs single-id calls) — **fixed in §1.4 below** by giving the resolver direct per-id dependencies mirroring `PlaybackSourceResolver`'s internals instead of wrapping it.
- **Verified**: `TimelineClip` already defines the interval as half-open `[timelineStartMs, timelineEndMs)` (see its own `TimelineInterval` doc comment: "Half-open timeline interval `[startMs, endMs)`"). `TimelineModel`/`VoicePool` **must** reuse this exact semantics rather than re-deciding it — resolves the boundary-flicker concern by pointing at an already-authoritative definition.
- **Verified**: `MixPlanClip` has fields exactly as documented, **plus** a `hasExplicitQueueItemId` flag already used by `QueueProvider._queueTimingClips()` to distinguish "the id was explicitly assigned" from "defaulted." No equivalent flag exists for `timelineStartMs`.
- **Verified regression risk (confirms Review 3's major finding)**: `QueueProvider._queueTimingClips()` (line ~433) sets `timelineStartMs: _firstTimelineStart(track) ?? existing?.timelineStartMs ?? 0` — i.e. **every clip the user hasn't manually dragged defaults to `timelineStartMs = 0`**. A persisted "Queue timing" mix plan for an ordinary 5+ track queue is therefore *not* the cascading, zero-overlap shape `TimelineModel.sequential` assumes; naively feeding it through `TimelineModel.fromMixPlan` would present 5 clips all starting at t=0 (depth 5) and the defensive depth-4 cap would **silently drop a track from ever playing**. Fixed structurally in §1.2/§3 via a dedicated `TimelineModel.fromQueuePlan` path — this is not cosmetic, it changes which constructor Phase 3 is allowed to call for the unified-queue case.
- `pubspec.yaml` declares `just_audio_background` (unused) and does **not** declare `audio_session` directly (it is a transitive dependency of `just_audio`). Phase 2 needs `audio_session` as a **direct** pubspec dependency for audio-focus coordination (§1.8) — this adds a manifest entry, not new native surface (just_audio already pulls it in transitively today).

## 1. Layer-by-layer design

### 1.1 TimelineClock (master transport)
Single source of truth for `positionMs`; **never** reads position from a Voice. A monotonic ticker anchored on `(lastKnownPositionMs, wallClockTimestamp)`, advanced by elapsed wall-clock time while `isPlaying`. Two tick rates: a UI-facing stream (~100-250ms, throttled coarser when no UI subscriber is active) and gain/drift cadences owned by `VoicePool` (§1.5/1.6), decoupled from each other (`gainUpdateIntervalMs` default 50ms vs. `driftCheckInterval` default 750ms, and drift-checking is skipped entirely for a single-clip timeline).

New: **buffering hold**. `holdForBuffering()`/`releaseHold()` + `isBufferingHeld`/`isBufferingHeldStream` let `VoicePool` pause position advancement (distinct from a user-initiated `pause()`, so the UI can show a buffering spinner rather than a paused icon) when the active voice set has zero ready voices — including, critically, the ordinary 1-voice case. Bounded by a hold timeout (15s default) that surfaces a recoverable stall error rather than hanging indefinitely.

Scrub remains a distinct sequence from a one-shot seek: `beginScrub()` → `updateScrub(ms)` (cheap, position-only, `isScrubbing == true`, no voice work) → `endScrub(ms)` (fires `scrubCommittedStream`, the only non-tick signal that triggers `VoicePool` to recompute the active set). `seek(ms)` is begin+update+end in one call.

The concrete `DefaultTimelineClock` implementation takes an injectable `DateTime Function() now` so unit tests can deterministically fast-forward virtual time (resolves A3-g) — this is a constructor detail of the implementation, not part of the abstract interface, so production code is unaffected.

### 1.2 TimelineModel / MixClip / GainEnvelope (pure, no IO)
Wraps the existing `TimelineClip` (half-open `[timelineStartMs, timelineEndMs)`, already the authoritative interval semantics — reused, not re-decided). `MixClip = {TimelineClip placement, GainEnvelope envelope, String audioSourceRef, String? queueItemId}`; **`audioSourceRef` defaults to `placement.trackId`** (fixed constructor bug from the draft — `audioSourceRef ?? placement.trackId`, with a Phase 1 regression test asserting exactly this).

`GainEnvelope{baseGainDb, fadeInMs, fadeOutMs, curve}` is a trapezoid shape whose `curve` is now **`FadeCurve.equalPower` by default** (constant-power crossfade, resolves the "hollow midpoint" concern with zero architectural cost — it's pure math inside `gainAt`, no new persisted field, curve is a client-rendering convention layered on the same `gainDb`/`fadeInMs`/`fadeOutMs` schema). `gainAt(localOffsetMs, clipDurationMs)` **hard-clamps its return value to `[0.0, 1.0]`** regardless of `baseGainDb`'s sign/magnitude — a safety net so `Voice.setVolume` is always called with a platform-valid value even if upstream authoring produced an out-of-range `gainDb`. Proportional fade-in/out shrink when their sum exceeds clip duration (same idiom as `TrimRange`/`TimelineClip`).

`TimelineModel` holds `List<MixClip>`, hard-caps `maxConcurrentVoices = 4` in every mutator, and provides `activeClipsAt`, `overlapDepthAt`, `dominantClipAt`, `canPlace`. Three named constructors:
- `TimelineModel.sequential(...)` — zero-overlap, flat-gain, the gapless-queue/single-song case.
- `TimelineModel.fromMixPlan(plan, {sourceDurationMsFor})` — for genuine shared/authored mixes. Defensive: drops any clip that would exceed depth 4 (backend does not enforce this yet — Open Decision #2), and clamps `gainDb` to `<= 0` (logs on clamp).
- `TimelineModel.fromQueuePlan(plan, {required List<String> trackOrder, required int Function(String) sourceDurationMsFor})` — **new**, for the specific "Queue timing" persisted-plan class that `QueueProvider` produces. Builds the `.sequential()` baseline from `trackOrder` first, then treats a persisted clip's `sourceStartMs/sourceEndMs/timelineStartMs` as an **explicit override only when it differs from that computed baseline value**; unedited (`timelineStartMs == 0`-by-default) clips fall back to the sequential placement instead of being taken literally. This is the load-bearing fix for the confirmed regression (§0) where an unedited persisted plan would otherwise collapse every clip to t=0 and lose tracks past the depth-4 cap.

`toMixPlanPayload({name})` is the inverse for `fromMixPlan`-shaped mixes.

### 1.3 Voice (wraps one `just_audio.AudioPlayer`)
One deck: `load(Uri, {initialLocalPositionMs})`, `seekLocal`, `setVolume(linearGain)` (caller-guaranteed `[0,1]` by `GainEnvelope`'s clamp), `setSpeed(rate)` (new — small ±2% nudges for sub-threshold drift absorption, see §1.6; never used for audible time-stretch), `play/pause`, plus:
- `release()`: stops and clears the loaded source but **keeps the native player instance warm** for fast pool reuse (explicit contract, resolves A1-h).
- `dispose()`: full teardown of the native player; called only when the pool itself shrinks or on app-level disposal.
- Drift diagnostics: `currentLocalPositionMs` (read only for drift-checking, never as a position source), `driftMs(expected)`, `resync(expected)`.
- `VoiceEvent{ready|buffering|stalled|completed|error}` stream so the pool reacts to slow network/decoder exhaustion without blocking the clock.

Constructed via an injected `VoiceFactory` (testable with `FakeVoice`, zero real audio in CI). The concrete implementation must configure a **short buffer window** (~10-15s min / ~20-30s max target on the underlying `AudioLoadConfiguration`/`LoadControl`) rather than ExoPlayer's single-deck-tuned defaults, to bound per-instance memory across 4-5 concurrent decks and avoid over-fetching data that frequent scrubbing throws away — called out as a required Phase 2 implementation detail (A2-e).

### 1.4 EngineAudioSourceResolver
Resolves a `MixClip.audioSourceRef` (a track id) to a playable `Uri`, reusing the **actual internals** `PlaybackSourceResolver` uses (local-artifact → cache → signed-URL), not the batch `resolveQueue`/`resolveTrack` API which doesn't fit a single-id engine call (A3-c fix). `DefaultEngineAudioSourceResolver` takes direct per-id dependencies: `LocalAudioArtifactResolver?`, `PlaybackCacheManager?`, `SignedAudioUrlService`, and a new `TrackAudioDescriptorProvider` seam that supplies the track-metadata map (`title`/`artist`/`duration`/etc.) `SignedAudioUrlService` needs, mirroring exactly what `PlaybackSourceResolver` already does per-track internally.

`warm(trackId, {required Set<String> protect})` — **`protect` is now required**, not optional; `VoicePool` always passes its current active-clip track ids, matching `PlaybackSourceResolver._scheduleWarm`'s existing eviction-safety pattern (A2-f fix).

New: a shared `SharedFetchGate` (injected into the resolver) caps **total concurrent network fetches** across streaming `Voice.load()` calls and speculative `warm()` calls (default budget e.g. 4), and de-prioritizes/defers speculative warms while ≥3 voices are actively streaming — resolves A2-b (no shared budget → up to ~8 simultaneous connections at the densest overlap, exactly when bandwidth matters most).

### 1.5 VoicePool + Mixer
Pool size = 4 active + `warmSpareVoices` (default 1) — sized for **steady-state**, not transient scrub spikes, because scrub-commit is now explicitly **release-before-acquire** (A2-d fix): clips leaving the active set are released to the pool *before* newly-active clips acquire voices. This is safe because a leaving clip's voice is already at gain 0 (its fade-out envelope reached silence before it's considered "left" — `activeClipsAt` uses the half-open interval, so a clip is only removed from the active set once its own window, including any fade-out tail, has ended), so releasing it first produces no audible artifact.

On every clock tick (normal playback) **and** on every `scrubCommittedStream` event (never during `isScrubbing`):
1. `active = timelineModel.activeClipsAt(clock.positionMs)` (≤4 by construction).
2. Stable diff against the current `clipId → Voice` map, tagged with a monotonically increasing **generation token** for this reassignment pass:
   - left → release to pool (already silent, see above);
   - still active → no reload; update `volume = envelope.gainAt(pos)` on the fast `gainUpdateIntervalMs` loop (decoupled from drift-checking, A1-e fix) and drift-check on the slower cadence;
   - newly active → acquire a (preferably pre-warmed) voice, `seekLocal`, muted `play()` **`voiceAttackLeadMs` (default ~120ms) before** its nominal activation instant so decoder/AudioTrack startup latency is absorbed before the envelope's fade-in is meant to become audible, then unmute exactly at the nominal instant per the envelope.
3. **Look-ahead preload**: for clips starting within `lookAheadMs` (default 4-6s), trigger `EngineAudioSourceResolver.warm(trackId, protect: activeTrackIds)` and `load()` a spare voice paused at the right offset. **A genuine 0-gap/0-fade boundary additionally schedules a boundary-precise one-shot timer** (not just the periodic tick) so gapless queue advance gets the same near-zero engineered crossfade treatment as an authored overlap, hiding voice-startup latency even when the plan specifies a literal 0ms fade (A1-d fix) — this is still "no special-casing" of the *musical* logic, only of the *scheduling precision* used to hide platform latency.
4. **Slow network mid-overlap**: if a look-ahead clip isn't ready by its `timelineStartMs`, the clock keeps advancing (per decision #1) as long as **at least one** active voice is healthy; the not-yet-ready voice joins late at gain 0 and fades in once ready. **If it is the only clip that would be active** (the ordinary 1-voice or fully-stalled-overlap case), `VoicePool` instead calls `clock.holdForBuffering()` so the transport itself pauses rather than skipping through silence (A2-a fix) — resumed automatically once ≥1 voice is ready, or surfaced as a recoverable stall after the hold timeout.
5. **Scrub commit**: recompute active set fresh at the destination. Sequence: release leaving clips (already silent) → `load`+`seekLocal` newly-active clips **in parallel**, bounded by `prepareTimeoutMs` (default 1500ms) → unmute/play whichever are ready at that point together; any voice still not ready fades in late using the **same** late-join mechanism as step 4 (A1-b/A2-c fix — no more "wait for all indefinitely"). Every async `load`/`seekLocal` callback checks its captured generation token against the pool's current generation before applying its result, so a stale callback from a superseded scrub is dropped rather than mutating a voice that's since been reassigned.
6. **Decoder exhaustion**: if acquiring a voice raises `VoiceEventKind.error` (e.g. platform decoder-session limit hit), the pool drops the current lowest-gain active voice to free a decoder rather than failing the whole activation (A2-g fix); flagged for on-device validation of real device decoder limits in Phase 2.
7. **Focus-driven pause**: `VoicePool` subscribes to `clock.isPlayingStream`; on transition to `false` (whether user-paused or focus-lost via `AudioFocusCoordinator` calling `PlaybackEngine.pause()`), every active voice is paused **in place** (not released) — no reassignment, no reload. On transition back to `true`, they resume `play()` directly; because the clock was held (not coasting) while paused, `expectedLocalMs` never advanced past reality, so **no corrective `seekLocal` is needed on resume** (A1-a fix — the stutter mechanism is eliminated, not just mitigated).

### 1.6 Drift/resync strategy
`expectedLocalMs = trimStartMs + (clock.positionMs - clip.timelineStartMs)`, recomputed fresh from immutable placement + one global position every check (never voice-to-voice compounding). Checked at `driftCheckInterval` (~750ms default, **skipped entirely when `timelineModel.isSingleClip`** — A2-h). Two-tier correction (A1-c fix):
- `|drift| <= 400ms` (default `speedNudgeThresholdMs`): absorbed with a brief `Voice.setSpeed` nudge (±2%, applied for well under a second) — imperceptible at that magnitude and duration, avoids the audible micro-mute a hard seek causes on VBR-encoded sources.
- `|drift| > 400ms`: escalates to a plain `seekLocal` (cheap on ExoPlayer, no reload) against the freshly recomputed expected position — reserved for large drift only, not routine jitter.

### 1.7 PlaybackEngine facade
Owns `TimelineClock` + `VoicePool` + current `TimelineModel`. Exposes `positionMsStream/durationMs/isPlaying`, `play/pause/seek/beginScrub/updateScrub/endScrub`, `loadMix(TimelineModel)`, `loadSequentialQueue(...)`, `nowPlayingStream` (`MixNowPlayingInfo`), and **new**: `clipCompletedStream` (`ClipCompletionEvent{trackId, clipId, wasSkipped}` — fires when a clip's own active window ends naturally during forward playback; `wasSkipped=true` when it left the active set via a seek/scrub jumping past it instead) and `dominantTrackIdChangedStream`. These two new streams are what let `PlaybackState`'s translation layer (§1.9) reconstruct the exact legacy surface `PlayRecorderService` depends on — and are a strict improvement over the old just_audio-`ProcessingState.completed`-only signal, which only reliably fired at the very end of the whole `ConcatenatingAudioSource` sequence rather than per mid-queue clip.

`skipToNextClip/skipToPreviousClip/skipToClipIndex`: for the sequential case (resolves A3-h), `skipToPreviousClip()` at the first clip **restarts it from zero** rather than no-op, matching today's `AudioPlayerService` behavior exactly. For a genuinely overlapping mix, "next/previous distinct `timelineStartMs`" remains the recommended default but is explicitly still a product call (Open Decision #6).

### 1.8 MixAudioHandler + AudioFocusCoordinator (audio_service / audio_session)
`AudioFocusCoordinator` (new, `client/lib/core/audio/audio_focus_coordinator.dart`, requires adding `audio_session` as a **direct** pubspec dependency — already a transitive dep of `just_audio`, so no new native surface) owns exactly one thing: on transient/permanent Android audio-focus loss or a "becoming noisy" (headphones unplugged) event, call `PlaybackEngine.pause()`; on focus regain, call `PlaybackEngine.play()`. It is deliberately **not** aware of voices/mixing — all the actual multi-voice coordination lives in `VoicePool`'s existing `clock.isPlayingStream` subscription (§1.5 step 7). A **minimal** `MixAudioHandler` (play/pause/seek passthrough, coarse metadata) ships in **Phase 2** alongside this, specifically so 4-voice overlap + screen-off/backgrounding get validated together rather than deferred to Phase 5 (A1-a fix).

The full `MixAudioHandler` (Phase 5) reports the mix as **one** `MediaItem`: `duration = totalDurationMs`, position = the global clock, title = dominant voice's title + `" · N layered"` when `activeVoiceCount > 1`. **`mixIdentity()`** (the `MediaItem.id`) returns the **dominant track's own id** whenever `activeVoiceCount == 1` (re-derived on every `dominantTrackIdChangedStream` event) — so plain single-track playback and gapless queue advance present to the OS exactly like today's per-track `MediaItem` continuity (Android Auto/CarPlay/art-cache keying all rely on this). Only when `activeVoiceCount > 1` does it switch to a distinct "layered mix" identity (A3-f fix). Every transport control, including the notification seekbar, calls back into the same `PlaybackEngine.seek()`.

Any Phase 5 file referencing both the app's `PlaybackState` (ChangeNotifier) and `audio_service`'s own `PlaybackState` type imports the latter as `import 'package:audio_service/audio_service.dart' as audio_service;` (A3-i fix, called out explicitly in Phase 5 scope).

### 1.9 PlaybackState translation boundary (the actual "unify" seam)
`PlaybackEngine` never imports `just_audio`/`audio_service` types — it only speaks `MixNowPlayingInfo`/`ClipCompletionEvent`/plain Dart. `PlaybackState` (which already owns exactly this translation job today, converting `AudioPlayerService`'s just_audio streams into its own public surface) is where the mapping happens:
- `positionStream`/`position` ← `Duration(milliseconds: engine.positionMs)` from `positionMsStream`.
- `currentMediaItemStream`/`currentItem` ← built from `nowPlaying.dominantTrackId` + `TrackMetadataProvider`, matching `PlaybackSourceResolver.buildLocalMediaItem`/`buildRemoteMediaItem`'s existing `MediaItem` shape.
- `playerStateStream` ← synthesized `just_audio.PlayerState`, with `processingState` driven by `clipCompletedStream` events for the **currently-armed** track (`wasSkipped == false`) instead of the old queue-wide `ProcessingState.completed`.

This is a from-scratch reimplementation of three streams' *producers*, but their **types and consumer-facing semantics are unchanged**, so `PlayRecorderService` requires zero code changes (A3-a fix) — Phase 3 adds a dedicated adapter test asserting exactly that: feed a `FakeVoice`-backed `PlaybackEngine` through a short/mid-queue/completed scenario and assert `PlayRecorderService` records the same play events it does today.

## 2. Data flow walkthroughs

**Play a single song**: `loadSequentialQueue([trackId])` → `TimelineModel.sequential` (1 clip, flat gain) → `play()` → clock ticks → Mixer activates the one clip on one voice. Same code path as a 4-track mix.

**Scrub the waveform mid-overlap**: `beginScrub()` → drag frames `updateScrub(ms)` (playhead moves, active voices silenced, no reassignment) → `endScrub(commitMs)` → Mixer recomputes `activeClipsAt(commitMs)` → releases leaving clips (already silent) → prepares newly-active clips in parallel bounded by `prepareTimeoutMs` → unmutes whichever are ready together, stragglers fade in late.

**Overlap begins during steady playback**: clip B enters the look-ahead window → warmed/preloaded on a spare voice, muted-primed `voiceAttackLeadMs` before its nominal instant → tick crosses the boundary → B's envelope ramps 0→base (equal-power) while A's fade-out ramps toward 0, both computed independently from `clock.positionMs` on the fast gain loop.

**Backgrounded mid-4-voice overlap, phone call arrives** (new walkthrough, resolves A1-a): `AudioFocusCoordinator` observes transient focus loss → calls `engine.pause()` → clock's `isPlayingStream` flips false → `VoicePool` pauses all 4 active voices in place (no release, no reload) → clock is held, so position does not advance during the call → call ends, focus regains → `AudioFocusCoordinator` calls `engine.play()` → clock resumes ticking from exactly where it left off → all 4 voices resume `play()` with **no corrective seek**, because nothing drifted while everything was uniformly held.

**Sole voice stalls on flaky network** (new walkthrough, resolves A2-a): the one active voice reports `stalled` → `VoicePool` sees zero ready active voices → calls `clock.holdForBuffering()` → transport stops advancing, UI shows a buffering indicator (not paused) → voice recovers and reports `ready` → `VoicePool` calls `clock.releaseHold()` → playback resumes from the position it was held at, with no skip-through-silence and no forward jump.

## 3. mix_plan persistence mapping

Two distinct load paths, both defensive against untrusted/legacy input:
- **`TimelineModel.fromMixPlan(plan, {sourceDurationMsFor})`** — genuine shared/authored mixes (product surface: "save playlist as mix", C17). Applies clips in payload order, drops any clip pushing depth past 4, clamps `gainDb` to `<= 0` (logs on clamp).
- **`TimelineModel.fromQueuePlan(plan, {trackOrder, sourceDurationMsFor})`** — the specific "Queue timing" plan class `QueueProvider` persists. Builds `.sequential()` from `trackOrder` first; a clip's `timelineStartMs`/source range only overrides the sequential baseline when it's **different** from what sequential placement would compute for that track — this is a heuristic ("was this actually edited"), documented and accepted for MVP, with the proper long-term fix (an explicit `isTimelineStartExplicit` flag on `MixPlanClip`, mirroring the existing `hasExplicitQueueItemId` pattern) tracked as Open Decision #9.

`toMixPlanPayload({name})` remains the exact inverse for `fromMixPlan`-shaped mixes. No backend schema changes required for the MVP envelope shape (curve is a client convention; gain clamp is a client-side load-time decision).

## 4. Notification mapping
See §1.8. `PlaybackState` and `MixAudioHandler` both derive from `PlaybackEngine.nowPlayingStream`/`dominantTrackIdChangedStream`, so mini player, full player, and lock screen can never disagree on "now playing." `mixIdentity()`'s dominant-track continuity rule (§1.8) means this is also true at the `MediaItem.id` level for OS integrations, not just the displayed title.

## 5. Testability
- `TimelineModel`/`GainEnvelope`/`MixClip`: pure Dart — depth-cap enforcement, equal-power/linear envelope math incl. proportional-shrink and the `[0,1]` clamp, half-open-boundary flicker regression, `MixClip.audioSourceRef` default, `fromMixPlan` vs `fromQueuePlan` round-trip fidelity (including the unedited-5+-track-queue regression case).
- `Voice` injected via `VoiceFactory`; `VoicePool`/Mixer diffing, generation-token staleness, release-before-acquire ordering under pool-exhaustion pressure, scrub-commit timeout/late-join, focus-driven pause/resume (no corrective seek), and starvation-hold are all testable with `FakeVoice` + the injectable-time `DefaultTimelineClock` — no real audio, no real delays.
- Only the walking skeleton (Phase 0) and the notification/on-device phases (Phase 2's real 4-voice + focus test, Phase 5's OS-integration polish) require manual on-device verification (consistent with the project's existing "file-backed/hardware audio is mobile-only, VM tests aren't the acceptance proof for that class" pattern).

## 6. Migration/unify risk
`PlaybackState`'s public field/method surface — including the three streams `PlayRecorderService` actually depends on — is preserved exactly (§1.9), backed by `PlaybackEngine` instead of `AudioPlayerService`. `QueueTimelineController` reimplements `setQueue/addToQueue/insertIntoQueue/removeFromQueue/skipToNext/skipToPrevious/skipToIndex/setShuffleMode/setLoopMode` on `TimelineModel.sequential`, replacing `ConcatenatingAudioSource`, with `skipToPrevious`'s restart-at-zero boundary and `setShuffleMode`'s "keep current item in place" contract explicitly specified and unit-tested (not left implicit, A3-h). `QueueProvider`'s existing "Queue timing" mix-plan auto-save is rebased onto **`TimelineModel.fromQueuePlan`** specifically (not `fromMixPlan`) — this is the concrete fix for the confirmed 0-default-timelineStartMs regression (§0, A3-e).
