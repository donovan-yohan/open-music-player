# Open decisions


1. **Backend overlap-cap enforcement** — `mix_plan_handlers.go` still does not reject >4-deep overlaps server-side; only the client engine does defensively on load (`fromMixPlan`/`fromQueuePlan`). *Recommend*: add a server-side sweep-line depth check (mirroring `TimelineModel.overlapDepthAt`) before "share a mix" is exposed to other users, since a shared/adversarial plan is otherwise untrusted input this client only partially protects itself from (dropped clips, not a rejected save). Not blocking for Phases 0-3; should land before Phase 5's cross-user sharing surface ships broadly.

2. **Voice pool sizing / on-device profiling** — `4 active + warmSpareVoices(1)` steady-state is architecturally sufficient even for scrub-commit (release-before-acquire ordering), but battery/CPU/memory on low-end Android is still unverified. *Recommend*: ship this default but require the Phase 2 on-device profiling pass on a low-end target before it's the shipped default; consider a device-tier-based lower cap as a fallback if profiling is unfavorable.

3. **Permanently unavailable clip mid-mix** (network failure / deleted track) — silent skip + debug log vs. inline non-fatal banner. *Recommend* silent skip + debug log for MVP (today's single-track `_playbackError` model doesn't fit a partial-mix failure); revisit UX with product once real usage data exists.

4. **`skipToNextClip`/`skipToPreviousClip` semantics for a genuinely overlapping (non-queue) mix** — resolved for the sequential case (§1.7: restart-at-zero at the start boundary). Still open for a dense overlapping mix, where "skip" is a weaker concept. *Recommend*: "jump to the next/previous distinct `timelineStartMs` value across all clips" (deduped) as the default, product-confirmed before Phase 5 ships any UI exposing skip in an overlap context.

5. **`TimelineClock.rate`** — resolved: fixed at `1.0` for MVP, documented as an intentional no-op, not half-implemented. Revisit only if/when a speed/time-stretch feature is scoped.

6. **Hard-seek drift threshold (`hardSeekDriftThresholdMs`) and speed-nudge threshold** — defaults proposed (150ms / 400ms) are engineering estimates, not yet validated against real ExoPlayer decoder-priming/Bluetooth-latency variance across devices. *Recommend*: treat as tunable constants and revisit after Phase 2's on-device 4-voice smoke test surfaces real voice-to-voice onset variance numbers.

7. **`fromQueuePlan`'s "was this actually edited" heuristic** — comparing a persisted clip's placement against the freshly-computed sequential baseline is a heuristic, not a guarantee: a user who manually drags a clip back to exactly where sequential placement would already put it is indistinguishable from "never edited," and a rare correctly-computed collision is possible for other permutations. *Recommend* for MVP: accept the heuristic (low product impact — worst case is a manual edit that coincidentally matches sequential ordering silently reverting, which is rare and low-severity). Proper fix: add an explicit `isTimelineStartExplicit`/`isPlacementExplicit` boolean to `MixPlanClip`, mirroring the already-precedented `hasExplicitQueueItemId` field, as a small, additive backend schema change — track as a fast-follow, not MVP-blocking.

8. **Decoder-exhaustion drop-lowest-gain fallback UX** — dropping the quietest active voice on a `VoiceEventKind.error` (hardware decoder-session limit) is a reasonable default but has no user-facing signal today (silent degrade). *Recommend*: ship silent for MVP (matches decision #3's philosophy), add a debug-log/telemetry hook so real-world frequency can be measured before deciding whether it needs a UI treatment.

9. **Equal-power vs. linear curve as a persisted "recommendation" vs. a hard client convention** — since curve shape is not persisted (client-rendering convention only), a future non-Flutter client (web, a hypothetical native client) rendering the same mix_plan with a different default curve would sound subtly different from this client despite reading identical data. *Recommend*: document the equal-power convention in the mix_plan API's shared documentation (not just this client's code) so any future renderer matches; not a schema change, a docs/consistency task.


# Residual risks


Residual risks after folding in all review fixes:

- **Voice-to-voice onset variance is still only approximately modeled.** The attack-lead pre-priming (`voiceAttackLeadMs`, default 120ms) and two-tier drift correction are engineering estimates, not measurements. Real ExoPlayer decoder-priming latency, VBR-codec seek-to-sync-point cost, and Bluetooth A2DP output-latency compensation vary by device/codec/output route in ways that can only be characterized on real hardware. Mitigation: Phase 2's on-device 4-voice test is explicitly scoped to surface these numbers, and the thresholds (Open Decision #6) are designed as post-launch-tunable constants, not hardcoded assumptions — but until that measurement happens, "near-zero engineered crossfade" is a design target, not a proven guarantee.
- **MediaSession/notification assumptions under real OS/head-unit variance.** Even with the `mixIdentity()` single-voice-continuity fix, Android's foreground-service/media-session model still fundamentally assumes one "now playing" item; the aggregate-mix presentation (dominant title + "N layered") when `activeVoiceCount > 1` bends that assumption and Bluetooth/Android Auto head units may render it inconsistently. This is explicitly flagged as unverified-until-tested in Phase 5, not silently assumed to work.
- **Heuristic `fromQueuePlan` edit-detection (Open Decision #7)** is a pragmatic MVP fix for a real regression, but it is a heuristic, not a structural guarantee — the proper fix (an explicit `isTimelineStartExplicit` flag) is deferred as a fast-follow, so a narrow edge case (a manual edit that happens to match the sequential baseline) can silently misclassify until that schema addition lands.
- **Backend overlap-cap enforcement remains client-only** (Open Decision #1) — a malicious or buggy non-Flutter client (or a future backend change) could still produce/share a >4-deep plan; this client's `fromMixPlan`/`fromQueuePlan` defensively drop the overflow clips rather than rejecting the whole plan, which is a safe but silent degrade (a shared mix could arrive missing tracks with no error surfaced to the recipient). Should not block MVP but is a real gap until server-side validation ships.
- **Cache-thrash / fetch-budget tuning under real usage.** `SharedFetchGate`'s default budget (e.g. 4 concurrent fetches, de-prioritize warms at ≥3 streaming voices) is a reasonable starting point but, like the drift thresholds, is untuned against real network conditions and cache eviction pressure from rapid scrubbing through a long mix. Flagged for Phase 2 implementation-time tuning, not a solved problem.
- **Hardware decoder-session limits are unverified across the actual target device matrix.** The drop-lowest-gain fallback (Open Decision #8) handles the failure gracefully but the *frequency* of hitting real decoder limits at 4-5 concurrent `AudioPlayer` instances on mid/low-tier Android hardware is unknown until Phase 2's on-device pass; if it turns out to be common, the "4-voice CDJ standard" product decision itself may need revisiting for a device-tier-based cap (out of scope for this design to decide, flagged for product/eng follow-up).
- **`PlaybackState` translation-shim parity (§1.9) is asserted, not yet proven**, until Phase 3's dedicated `PlayRecorderService` adapter test exists. The design is confident this preserves exact consumer-facing behavior (and in the mid-queue-short-track case, actually improves on it via true per-clip `ClipCompletionEvent`s instead of queue-wide `ProcessingState.completed`), but this is a claim to be verified by that specific Phase 3 test, not an already-verified fact.
