
**Phase 0 — Walking skeleton (on-device feasibility proof).**
Scope: full `TimelineClock` (incl. injectable time source + `holdForBuffering`/`releaseHold` plumbing, even if unused yet) + thin `Voice` wrapper + `VoicePool` hardcoded to 2 voices, no `TimelineModel` persistence, no notification. Deliverable: a debug-only screen (behind a debug menu, not in main nav) that authors a 2-track overlap by hand (track B starts at track A's midpoint, both with a 6s equal-power fade), plays, and drags one slider that scrubs both voices via `clock.seek`. Proves: 2 simultaneous just_audio streams both audible with independent gain; global scrub repositions both; background survival via a throwaway minimal audio_service handler pointed at the clock just for this proof (full focus/notification handling is Phase 2, not here). Verified by: manual on-device play test (matches the project's existing "file-backed/hardware audio is mobile-only" acceptance pattern), plus `flutter test`s for `DefaultTimelineClock`'s tick math using the injectable time source (deterministic, no real delays) and a `FakeVoice` `VoicePool` diff test as CI regression coverage.

**Phase 1 — TimelineModel (pure), fully unit-tested, no audio/UI wiring.**
Scope: `GainEnvelope` (equal-power default + linear, `[0,1]` clamp), `MixClip` (with the fixed `audioSourceRef ?? placement.trackId` default), `TimelineModel` (cap-of-4 `canPlace`, `activeClipsAt`/`overlapDepthAt` using the half-open `[start,end)` interval already authoritative in `TimelineClip`, `dominantClipAt`, `.sequential`, `.fromMixPlan` with the `gainDb <= 0` clamp, and **`.fromQueuePlan`** with its sequential-baseline-diff override logic). Deliverable: a fully unit-tested pure package, zero dependency on just_audio/audio_service. Verified by `flutter test`: `MixClip(placement: p).audioSourceRef == p.trackId` regression test; random-placement property tests never exceed depth 4 after `canPlace`'s gate; envelope math incl. proportional-shrink and clamp boundary; off-tick sampling produces no active-set flicker at a clip boundary; `fromMixPlan`/`toMixPlanPayload` round-trip fidelity; **`fromQueuePlan` regression test loading a real unedited 5+ track "Queue timing" plan (all persisted `timelineStartMs=0`) and asserting every track still plays in order**, not dropped past the depth-4 cap.

**Phase 2 — Full engine: VoicePool/Mixer + PlaybackEngine facade + minimal focus-aware notification, generic-N with real cap=4.**
Scope: look-ahead preload + `PlaybackCacheManager` integration via the corrected `DefaultEngineAudioSourceResolver` (direct per-id deps, `SharedFetchGate`, required `protect` set on `warm()`), drift/resync (speed-nudge + escalation-to-seek two-tier strategy), decoupled gain-update loop, scrub-commit with `prepareTimeoutMs`/generation tokens/release-before-acquire ordering, gapless boundary-precise timer + attack-lead pre-priming, starvation-hold (`clock.holdForBuffering`), decoder-exhaustion drop-lowest-gain fallback, per-Voice short buffer-window configuration. **Also in this phase (pulled forward from the original Phase 5 per review): `AudioFocusCoordinator` + a minimal `MixAudioHandler`** (play/pause/seek passthrough, coarse metadata) wired into the Phase 0 debug screen, and `audio_session` added as a direct pubspec dependency. Deliverable: `PlaybackEngine` logic passes entirely against `FakeVoice` (VM tests), plus the debug screen extended to 4 real overlapping tracks **with an on-device screen-off/phone-call test** verifying uniform pause/resume with no corrective-seek stutter. Verified by: `flutter test` for assignment diffing, cap enforcement, scrub timeout/late-join, generation-token staleness rejection, pool-exhaustion-under-release-before-acquire, focus-pause/resume-no-seek, starvation-hold/release, decoder-exhaustion fallback; manual on-device 4-voice overlap + backgrounding/phone-call smoke test.

**Phase 3 — Unify: rebase `PlaybackState` onto `PlaybackEngine`.**
Scope: replace `AudioPlayerService`/`ConcatenatingAudioSource` inside `PlaybackState` with `PlaybackEngine` + the `PlaybackState` translation shim (§1.9) synthesizing `positionStream`/`currentMediaItemStream`/`playerStateStream` from `nowPlayingStream`/`clipCompletedStream`/`dominantTrackIdChangedStream`; new `QueueTimelineController` (queue ops on `TimelineModel.sequential`, with the restart-at-zero and keep-current-item-on-shuffle contracts specified and tested); rebase `QueueProvider`'s "Queue timing" mix-plan auto-save load path onto **`TimelineModel.fromQueuePlan`** (not `fromMixPlan`); update `main.dart` wiring; delete `audio_player_service.dart` and the unused `just_audio_background` pubspec entry. `PlaybackState`'s public surface (including the three streams `PlayRecorderService` depends on) is unchanged in type/semantics. Also: on-device gap/overlap measurement of gapless boundaries against the pre-migration `ConcatenatingAudioSource` baseline. Verified by: existing `playback_state_test.dart` adapted to a `FakeVoice`-backed `PlaybackEngine`; a dedicated `PlayRecorderService` adapter test feeding short/mid-queue/completed scenarios through the new translation shim and asserting identical recorded-play behavior to today; gapless-boundary and resume/`restore()` parity tests; manual on-device smoke pass (play/pause/seek/skip/resume) via the `run`/`verify` skills as the highest-risk regression point.

**Phase 4 — Waveform wiring.**
Scope: rebind `StackedWaveformTimeline`'s playhead to `PlaybackEngine.positionMsStream` (live, not synthesized), rewire drag gestures to `engine.beginScrub/updateScrub/endScrub`, render real `TimelineModel` clips/overlap bands with gain-derived visual feedback. Verified by: updated widget tests plus new tests asserting drag calls the scrub API.

**Phase 5 — Full notification polish + cross-device verification.**
Scope: complete `MixAudioHandler` (dominant-voice title + "· N layered", `mixIdentity()`'s single-voice-continuity rule, art), throttled `PlaybackState`-derived pushes to the OS transport, `main.dart` wiring using `import 'package:audio_service/audio_service.dart' as audio_service;` in any file referencing both `PlaybackState` types. Verified by: manual on-device (mobile-only) verification of lock-screen/notification transport controls scrubbing the whole mix, title updating correctly during an overlap window, and `mixIdentity()` continuity across a gapless queue transition on at least one Bluetooth/Android Auto-class target (unverified-until-tested per Risks).
